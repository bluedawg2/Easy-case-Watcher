<objective>
Fix the Windows psycopg async event loop incompatibility. On Windows, asyncio
defaults to `ProactorEventLoop`, but the async psycopg 3 driver (used by
SQLAlchemy via `postgresql+psycopg://`) only works on a `SelectorEventLoop`.
Standalone async entry points — `brm.seed`, `brm.run_pipeline`, the Uvicorn dev
server, and any future Procrastinate worker — crash at first DB connection with:

  sqlalchemy.exc.InterfaceError: (psycopg.InterfaceError) Psycopg cannot use the
  'ProactorEventLoop' to run in async mode.

Purpose: Make every async entry point use a psycopg-compatible event loop via a
single shared startup guard, instead of patching each `asyncio.run` call site.

Output: A Windows-guarded event loop policy guard at the top of
`src/brm/__init__.py`.
</objective>

Decision: the guard belongs in `src/brm/__init__.py` (the package __init__),
NOT db.py. Rationale:
- Every async entry point imports a `brm.*` module, so the package __init__.py
  always executes first. Confirmed entry points:
    - `brm/seed.py`        → imports `brm.db` → triggers `brm/__init__.py`
    - `brm/run_pipeline.py`→ imports `brm.db` → triggers `brm/__init__.py`
    - `brm/main.py` (Uvicorn `brm.main:app`) → triggers `brm/__init__.py`
    - future Procrastinate worker (Phase 4) → will import `brm.*`
- `__init__.py` runs at import time, BEFORE any `asyncio.run()` / Uvicorn / worker
  creates an event loop — so the policy is set before a loop exists.
- db.py would also work today, but a future non-DB async entry point might not
  import it; the package __init__ is the durable single home.
