<objective>
Build the AI slice: a single-turn Anthropic Messages call that turns a `detected` Change's verbatim rulemaking-entry diff into the D-05 structured summary (headline + What changed / Where / To whom / For what cases), persists it as JSONB (D-09), attaches the server-set "not legal advice" label (D-08), and advances the Change to `processed`. The prompt enforces concise output (D-06) and the no-speculation / no-advice guardrails (D-07).

Context — what changed from the reviewed plan: three review findings reshape this plan. (1) Unverified SDK call (#2): all four reviewers flagged `client.messages.parse(output_format=...)` as unverified and likely wrong (it resembles OpenAI SDK syntax) — this plan now opens with a discrete `[BLOCKING]` SDK-verification task that confirms the exact structured-output surface of the installed `anthropic` 0.103.1 against the real SDK, with a documented fallback, BEFORE any summarize code is written. (2) Validation theater (#7): the previous plan's offline fixtures could pass forever while live ingestion fails — a `test_live_integration` task is added that fetches the real FRBP source and asserts the observed schema before phase completion (compatible with D-04: live runs in parallel). (3) AI-input payload (Gemini): because plan 01-02 now uses a real FRBP rulemaking source, the diff fed to the AI is actual rulemaking/amendment entry text, not a news blurb — the prompt summarizes that rulemaking entry text; following links to full PDF amendment text is explicitly Phase 3 scope and out of bounds here.

Purpose: This is the slice that makes a change human-readable. After this plan a developer can run a `detected` Change through the summarizer and observe a `processed` Change with a structured summary — the content the reviewer reads in plan 01-05.
Output: SDK-verification record, `ChangeSummary` Pydantic schema, the `summarize` wrapper using the verified structured-output call, pipeline `run_summarize` step, admin CLI, AI tests with a mocked SDK, and a live-integration verification.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| diff text → Anthropic API | Verbatim untrusted source-derived rulemaking text is sent to the LLM |
| Anthropic response → database | Model output persisted as JSONB on the Change |
| live FRBP source → live-integration test | A real external government source is fetched during verification |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | Prompt injection — untrusted feed/diff text instructing the model | mitigate | The diff is passed inside a clearly-labelled user-content data block, never as instructions; the fixed `SYSTEM_PROMPT` pins the task; output is schema-constrained by the verified structured-output call so injected free-form instructions cannot reshape the response; the summary is human-reviewed (plan 01-05) before it can reach the pull API. Full prompt-injection hardening is a later-phase concern; Phase 1 notes the surface and applies these baseline controls |
| T-04-02 | Spoofing | The model omitting or altering the "not legal advice" label | mitigate | The label is a server-side module constant attached in `run_summarize`; it is NOT a `ChangeSummary` field — the model cannot omit or reword it (RESEARCH Pitfall 3; D-08; AI-06) |
| T-04-03 | Information Disclosure | `ANTHROPIC_API_KEY` leaking into source or logs | mitigate | Key loaded via `pydantic-settings` from a git-ignored `.env`; never logged; offline tests mock the SDK and run without the key; the SDK-verification (Task 1) and live-integration test use the env key only |
| T-04-04 | Denial of Service | The blocking AI call hanging and stalling the event loop | mitigate | `run_summarize` runs `summarize` via `asyncio.to_thread` under `asyncio.wait_for(timeout=...)`; a timeout transitions the Change to `summary_failed` rather than hanging the server (review finding #4) |
</threat_model>
