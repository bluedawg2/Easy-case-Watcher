<objective>
Stand up the Walking Skeleton's foundation: a uv-managed Python 3.12 / FastAPI backend project, a local Postgres 16 via Docker, and the three core domain models (Source, Snapshot, Change) with a hand-written Alembic migration. This is the first slice — after this plan a developer can run a test suite against a real database; the FRBP source is seeded in plan 01-02.

Context — what changed from the reviewed plan: four review findings touch this plan. (1) Async test transaction strategy + `brm_test` DB (#18): the previous plan handwaved "transaction + rollback per test" and left the test-DB-creation mechanism unchosen — this plan specifies the EXACT async SQLAlchemy SAVEPOINT/nested-transaction fixture pattern and picks ONE `brm_test` creation mechanism (a Postgres init script). (2) Migration autogenerate drops CHECK constraints (#19): the initial migration is now hand-written, not autogenerated, so the status CHECK constraint and indexes are guaranteed. (3) `health_status` typed (#20): `health_status` is a typed value set with `healthy` set on success (the setting-on-success happens in plan 01-03's `run_ingest`). (4) Index on `Change.status` (#21): `idx_change_status` is added to the hand-written migration.

Purpose: Every subsequent slice (fetch, detect, summarize, review/API) reads and writes these tables. The lifecycle state machine and the tri-date Change model are established here so no later plan re-litigates the schema.
Output: Runnable backend project, Postgres container, three migrated tables, lifecycle guard, passing model + lifecycle tests.
