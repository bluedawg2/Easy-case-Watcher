# Phase 1 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope of This Contract

Phase 1 has exactly **one UI surface**: the **internal review-queue SPA** (ROADMAP plan 01-05). It is the
human-review tier the whole vertical slice exists to prove. The pull delivery API (plan 01-05) is a
machine-to-machine JSON endpoint and has no UI.

**Screens in scope:**
1. **Queue list** — pending changes (`status = processed` / `in_review`), one row per Change, scannable.
2. **Change detail** — expandable per-row detail (D-09): AI summary structured fields, side-by-side diff,
   source link, snapshot view, and the approve / edit / reject action area.

**Audience:** product-team reviewers (internal). Not customer-facing. Unauthenticated internal tool by
explicit decision (CONTEXT.md "Claude's Discretion"; reviewer identity is a free-text `reviewer_name`
field only — full auth/attribution is Phase 8, AUDIT-02). Do not build login.

**Out of scope this phase:** classification badges, severity badges, confidence score, extracted-field
editing, effective-date calendar — those arrive in Phases 5/6. The only AI output to render/edit in
Phase 1 is the structured summary (D-11).

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn/ui (locked in CLAUDE.md — "copy-in components, no heavy framework lock-in") |
| Preset | New York style, `neutral` base color, CSS variables enabled. `components.json` to be created by `npx shadcn init` during plan 01-05 when the `web/` Vite package is scaffolded — `shadcn_initialized: false` is correct *now*; the checker should re-verify after 01-05. |
| Component library | Radix UI primitives (shadcn/ui default underlay) |
| Icon library | `lucide-react` (shadcn/ui default; install with the scaffold) |
| Font | Inter (UI sans) for all chrome and summary text; `ui-monospace` system mono stack for diff text and the raw snapshot view |

**Why New York style:** the review queue is a dense, table-driven internal tool — New York's tighter
spacing and smaller default control sizes suit data review better than the rounder Default style.

**shadcn components this phase needs** (all from the official registry — no third-party registries):
`table`, `badge`, `button`, `dialog`, `textarea`, `input`, `label`, `card`, `collapsible`,
`skeleton`, `sonner` (toast), `alert`.

## Spacing Scale

Declared values (multiples of 4 — 8-point scale):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps, badge inner padding |
| sm | 8px | Compact spacing inside table cells, between stacked summary fields |
| md | 16px | Default gap between controls; queue row vertical padding; card padding |
| lg | 24px | Detail-panel section padding; gap between summary block and diff block |
| xl | 32px | Page content gutter; gap between page header and queue table |
| 2xl | 48px | Empty-state vertical centering block |
| 3xl | 64px | Reserved — not used in Phase 1's single-page layout |

Exceptions: none. Diff-viewer internal spacing is owned by `react-diff-viewer-continued` and is not
re-themed in Phase 1 — accepted as a library default.

## Typography

Exactly 4 sizes, exactly 2 weights (regular 400, semibold 600).

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Display | 28px | 600 | 1.2 — page title ("Review Queue") |
| Heading | 20px | 600 | 1.2 — AI summary headline (D-05 one-line headline), detail-panel section labels |
| Body | 16px | 400 | 1.5 — summary detail block (what changed / where / to whom / for what cases), reviewer copy |
| Label | 14px | 600 (table headers) / 400 (cell text, captions) | 1.4 — table headers, badges, timestamps, form labels |

Diff text and the raw-snapshot view render in the monospace stack at 14px / 1.5 — diff legibility
overrides the body scale; this is the one place mono is used.

## Color

Light-mode-only for Phase 1 (internal tool; dark mode is not in scope). 60/30/10 split.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#FFFFFF` | Page background, table surface, detail panel |
| Secondary (30%) | `#F4F4F5` (zinc-100) | Page header bar, table header row, expanded detail-panel background, card surfaces |
| Accent (10%) | `#1D4ED8` (blue-700) | Primary "Approve" button, the active/focused row indicator, source-link affordance |
| Destructive | `#B91C1C` (red-700) | "Reject" action and its confirmation dialog only |

**Accent reserved for:** the primary **Approve** button, the source/snapshot link text, and the
keyboard-focus / selected-row ring. Nothing else. Secondary controls ("Edit", "Cancel", "View
snapshot") use shadcn `outline`/`ghost` button variants — neutral, not accent.

**Diff colors:** addition/removal coloring inside the diff is owned by `react-diff-viewer-continued`
(its conventional green/red). This is a semantic exception to the 10% rule — diff +/- coloring is
information, not decoration — and is not counted against the accent budget.

**Status indication:** Phase 1 has few statuses; render `detected` / `processed` / `in_review` /
`verified` / `rejected` as neutral shadcn `badge` (`secondary` variant) with the status text — not as
colored chips. Color-coded severity badges are deliberately deferred to Phase 5.

## Copywriting Contract

Tone: factual, neutral, non-advisory — mirrors the project's "report facts, not legal advice" stance.

| Element | Copy |
|---------|------|
| Page title | `Review Queue` |
| Primary CTA | `Approve change` |
| Secondary actions | `Edit summary` · `Reject` · `View snapshot` |
| Empty state heading | `No changes waiting for review` |
| Empty state body | `When a monitored source changes, detected changes appear here for review. Nothing is pending right now.` |
| Loading state | shadcn `skeleton` rows in the table — no spinner, no copy |
| Error state (queue load failed) | `Couldn't load the review queue. Check that the API is running, then retry.` — with a `Retry` button |
| Error state (review action failed) | `That action didn't go through. Your change wasn't saved — try again.` — surfaced as a `sonner` toast; the row stays pending |
| Success (approve) | Toast: `Change approved and verified.` |
| Success (reject) | Toast: `Change rejected.` |
| Reject confirmation (destructive) | Title: `Reject this change?` — Body: `Rejecting marks this as not a real rule change. It will not be sent to the consuming product and will leave the queue. This can't be undone in Phase 1.` — Confirm button: `Reject change` (destructive variant); Cancel: `Keep in queue` |
| Effective-date field label | `Effective date` — helper: `The calendar date the change takes effect. Read it from the diff or source. Required to approve.` |
| Not-legal-advice label | `Informational summary — not legal advice. Generated by AI from the source diff.` — rendered statically beneath every AI summary; **never** model-generated (AI-06; RESEARCH Pitfall 3) |

**Destructive actions this phase:** exactly one — **Reject**. It is confirmed via a shadcn `dialog`
(the copy above). Approve and Edit are not destructive and need no confirmation. There is no delete.

**Copy rules for the checker:**
- The not-legal-advice label text is fixed — it is a server/UI constant, not editable, not AI-output.
- "Approve" requires a valid `Effective date` and a non-empty headline; if missing, the Approve button
  is disabled and the field shows inline validation: `Enter a valid effective date to approve.`
- Edit mode edits only the AI summary fields (D-11) — headline + the four content fields. No other
  field is editable in Phase 1.

## Layout & Interaction Notes

For the planner/executor — interaction contract, not visual styling:

- **Single-page layout.** Page header (title + queue count) over a full-width `table`. No sidebar, no
  routing beyond the one page — there is one screen.
- **Queue row (D-09):** columns — Source, Headline (AI summary one-liner, truncated), Detected date,
  Status badge. Clicking a row toggles a `collapsible` detail panel inline beneath it (expandable
  per-row detail — the pattern the user pictured in CONTEXT.md "Specific Ideas").
- **Detail panel** contains, top to bottom: structured AI summary (headline as Heading; what-changed /
  where / to-whom / for-what-cases as labeled Body blocks), the not-legal-advice label, the
  side-by-side diff (`react-diff-viewer-continued`), a source link + `View snapshot` affordance, the
  `Effective date` input, and the action row (`Approve change` / `Edit summary` / `Reject`).
- **Edit mode** swaps the summary display blocks for `textarea`/`input` fields in place; `Save` and
  `Cancel` replace the action row until saved.
- **Data fetching:** TanStack Query owns queue state; approve/edit/reject are mutations with
  `onSuccess` query invalidation (RESEARCH code example). Optimistic UI is allowed but the row must
  visibly revert on a failed mutation (pairs with the review-action error toast).
- **Accessibility:** all actions reachable by keyboard; the focus ring uses the accent color; the
  collapsible toggle has an accessible label; diff +/- regions are not color-only (the library
  prefixes lines with +/− glyphs — keep that on).
