# Walking Skeleton — Bankruptcy Rule Monitor

**Phase:** 1
**Generated:** 2026-05-21
**Updated:** 2026-05-21 (revised after cross-AI review — `--reviews` replan)

## Capability Proven End-to-End

A real change in the verified U.S. Courts FRBP rulemaking source can be detected, AI-summarized, reviewed and approved by a human in a browser, and pulled by "the other product" through an authenticated read-only API — exercising the full ingest → detect → AI → review → deliver stack.

## Architectural Decisions

| Decision | Choice | Rationale |
|---|---|---|
| Backend language | Python 3.12 | Strongest scraping / PDF / Anthropic-SDK ecosystem; locked in CLAUDE.md |
| API framework | FastAPI 0.136.1 + Uvicorn 0.47.0 | Async-native (matches async fetch + LLM I/O), automatic OpenAPI schema for the consuming product, Pydantic v2 validation |
| ORM + migrations | SQLAlchemy 2.0.49 (async) + Alembic 1.18.4, psycopg 3 driver | Async ORM pairs with FastAPI; versioned reviewable migrations. The initial migration is HAND-WRITTEN (not autogenerated) so the CHECK constraints and indexes are guaranteed present |
| Data layer | PostgreSQL 16, run via Docker Compose locally | Single datastore for relational data + JSONB (LLM summary) + future job queue; Docker because Postgres is not installed natively. A `brm_test` database is created on the same container by a Postgres init script |
| Lifecycle `status` storage | VARCHAR + CHECK constraint (NOT a native PG enum) | The taxonomy provably gains values in Phases 5/6; VARCHAR+CHECK is trivially extensible vs awkward `ALTER TYPE` migrations (RESEARCH Pitfall 5 / Open Question 1) |
| Lifecycle states (Phase 1) | `detected → processed → in_review → verified` / `rejected`, plus `summary_failed` and an `in_review → in_review` self-loop | The Phase-1 subset; `summary_failed` is an operational-safety state so an AI-summary failure is never invisible; the `in_review` self-loop lets a reviewer re-edit a summary more than once. `classified` / `pending-effective` / `active` / `superseded` are Phases 5/6 |
| Source health | Typed `health_status` value set (`unknown` / `healthy` / `failed`) with a CHECK constraint; set to `healthy` on a successful fetch and `failed` on a FETCH_FAILED | A typed state, not a free-form string; a successful fetch moves it off `unknown` so a healthy source is distinguishable from a never-checked one |
| AI integration | Anthropic Python SDK 0.103.1, used directly — structured-output call (exact surface VERIFIED against the installed SDK before any summarize code is written), with a `messages.create()` + manual-JSON-parse fallback | A few fixed single-turn operations; no agent framework (CLAUDE.md "What NOT to Use"). The exact structured-output API is confirmed by a blocking verification task — not assumed |
| AI call execution | The blocking SDK call runs off the async event loop via `asyncio.to_thread` under an `asyncio.wait_for` timeout | A synchronous SDK call run inline in an async function would freeze the event loop and stall the API server |
| Ingestion seam | `SourceAdapter` Protocol with one `FrbpSourceAdapter` (httpx 0.28.1; `feedparser` for a feed source OR a thin `selectolax` HTML-list parse for a rulemaking-page source) | Establishes the adapter seam now; the feed-vs-page parse is chosen per the verified source. Phase 2/3 add the general Html/Pdf adapter framework with no seam change. The Phase-1 thin HTML parse, if used, is a property of this one source's adapter — NOT the Phase-2 general scraping framework |
| Background processing | None — pipeline functions invoked directly via a `python -m brm.run_pipeline` admin CLI | Procrastinate is locked in CLAUDE.md but its first use is Phase 4; standing up a worker now is roadmap scope creep. The `Source.polling_cadence` column exists and is seeded `daily` but is intentionally NOT yet driven by any scheduler — a known, documented Phase-1 limitation |
| Review UI | React 19 + Vite 6 + TypeScript SPA in `web/`, shadcn/ui + Tailwind, TanStack Query 5, react-diff-viewer-continued 4.2.2 | Internal tool; master/detail layout (UI-SPEC Variant B). React 19 compatibility of the diff viewer is verified during plan 01-05; pinning React 18 is the documented fallback |
| Auth | Minimal shared API-key middleware on both the review and pull routers (`X-API-Key` header from config); reviewer identity is a free-text `reviewer_name` field | The API key is the minimal control that closes the unauthenticated-access gap and satisfies the ASVS L1 threat model. Full per-tenant auth + reviewer-identity audit trail is Phase 8 (AUDIT-02) — do not over-build |
| Integration seam | A single authenticated read-only FastAPI `GET /changes` pull endpoint, with a `since`-cursor for incremental sync; no shared database | API-07 — the monitor and "the other product" stay decoupled. The `since` cursor is the minimal incremental-sync contract; full filtering and as-of-date temporal queries are Phase 7 |
| Directory layout | `src/brm/` package (models/ schemas/ ingest/ detect/ ai/ api/); `tests/`; `web/` SPA; `alembic/` migrations | Per RESEARCH § Recommended Project Structure |
| Multi-tenancy seam | Nullable `tenant_id` column on tenant-scoped tables (source, change) from day one | Cheap insurance per CLAUDE.md — "architected to become a sellable add-on". In Phase 1 the column is intentionally un-scoped fiction (no per-tenant auth yet) — a known, deliberate decision, not a defect |
| Snapshot storage | Append-only snapshot rows, all retained | Phase 1 has one source with ~annual real changes, so volume is trivial. A retention / compression / content-addressing strategy will be required at the eventual 140+ sources — a documented later-phase concern, not built here |

## Source Definition (revised after review)

The unanimous cross-AI review finding was a source/domain mismatch: the earlier plan seeded the generic `uscourts.gov/news/rss` Judiciary News feed while the declared subject layer is FRBP. Resolved:

- **D-01 stays locked** — the Phase-1 subject layer is the Federal Rules of Bankruptcy Procedure (FRBP).
- The seeded source is the **verified, authoritative FRBP rulemaking source** — the U.S. Courts "Pending Rules & Forms Amendments" / rulemaking area — identified and verified by a live fetch in plan 01-02 (Task 1, a blocking checkpoint that produces `01-02-SOURCE-VERIFICATION.md`). It is NOT the generic Judiciary News feed.
- RESEARCH established the rulemaking page carries no RSS/Atom feed. If the verified source has no clean feed, the Phase-1 `FrbpSourceAdapter` fetches the rulemaking page directly with a thin HTML-list parse — a property of this one source's adapter behind the `SourceAdapter` seam, distinct from the Phase-2 general HTML-scraping framework.
- A real captured live snapshot of the verified source is the happy-path fixture (`tests/fixtures/frbp_source_v1.captured`); a modified copy (`frbp_source_v2.captured`) carries the simulated change-under-test. A live-integration test (plan 01-04) fetches the real source and asserts the observed schema before phase completion.

## Out of Scope (Deferred to Later Slices)

- HTML scraping and PDF text extraction as a general framework (Phases 2–3) — the Phase-1 thin per-source HTML parse, if the verified FRBP source is a page, is not the general framework
- Following an RSS/page link to the full PDF amendment text (Phase 3) — the Phase-1 AI input is the rulemaking-entry text only
- Additional sources — districts, state exemption rules, the full national layer (Phases 2–4)
- Source health monitoring beyond the typed `health_status`, staleness/layout-drift alerting, cross-channel deduplication (Phase 2)
- Adaptive polling cadence, the Procrastinate scheduler/worker, AI confidence scoring (Phase 4) — Phase 1 is manually triggered via the admin CLI
- Full 3-axis AI taxonomy classification and structured field extraction (Phase 5)
- Confidence-gated tiered routing, auto-publish, correction/retraction (Phase 5)
- Effective-date scheduler, pending-effective/active states, relative-date resolution, effective-date calendar (Phase 6)
- Temporal as-of-date queries, API contract versioning, Official Forms supersession (Phase 7) — Phase 1 ships only the minimal `since`-cursor incremental contract
- Immutable audit trail, AI reproducibility records, stable internal IDs, real per-tenant authentication and reviewer attribution (Phase 8)
- Snapshot retention / compression / content-addressing (later phase) — Phase 1 retains all snapshots; trivial volume at one source
- AI date-hint for effective date — a quick win deferred per CONTEXT.md Deferred Ideas
- Lifecycle taxonomy states `classified` / `pending-effective` / `active` / `superseded` (Phases 5/6)
