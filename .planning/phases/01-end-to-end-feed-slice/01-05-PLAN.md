<objective>
Build the final slice that closes the loop: a FastAPI review-queue backend (list pending changes, approve / edit / reject — all three per D-10), a read-only pull delivery API exposing only verified changes with a since-cursor for incremental sync, minimal API-key auth on both routers, and the React + Vite review-queue SPA the reviewer uses. After this plan the whole vertical slice is demonstrable — a detected, AI-summarized change can be reviewed in a browser and pulled by "the other product" via the API.

Context — what changed from the reviewed plan: three review findings add work here. (1) No auth on review/pull APIs (#12): a new `src/brm/auth.py` adds a shared single-key API-key dependency applied to BOTH routers — an unauthenticated request is rejected; this also satisfies the ASVS L1 threat model. (2) Pull API has no incremental sync (#13): the pull API gains a `since`/updated-after cursor parameter and a stable secondary ordering so the consuming product fetches only what changed (API-01/API-07); full filtering and temporal queries stay Phase 7. (3) react-diff-viewer-continued React 19 compat (#17): Task 3 adds an explicit verification step that the diff viewer renders under React 19 in the Vite project, with pinning React 18 documented as the fallback.

Purpose: This is the human-review tier the entire slice exists to prove, plus the API-07 integration seam. It completes the Walking Skeleton: one real UI interaction wired to the API, and a machine-readable, authenticated, incrementally-syncable pull endpoint.
Output: Auth dependency, review + pull API routers, the FastAPI app, the React review SPA (master/detail layout per UI-SPEC Variant B), passing auth, review-API, pull-API, and component tests.
</objective>

## Trust Boundaries

| Boundary | Description |

| "the other product" → pull API | The only integration seam; machine caller crosses here; gated by API key |
| API responses → review SPA | Feed/AI text rendered in the browser |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Information Disclosure | Pull API leaking non-verified changes to "the other product" | mitigate | `GET /changes` filters `status == verified` at the query level; `test_only_verified` integration test enforces it; no write endpoint and no DB credentials exposed (API-07) |
| T-05-02 | Tampering | SQL injection via reviewer input, the `since` parameter, or any filter parameter | mitigate | All endpoints use SQLAlchemy 2.0 parameterized ORM queries; `since` is typed `datetime` and validated by FastAPI; request bodies validated by Pydantic models; no string-formatted SQL |
| T-05-03 | Tampering | Stored XSS — feed/diff/AI summary text rendered in the SPA | mitigate | React escapes JSX by default; `web/src/` uses no `dangerouslySetInnerHTML`; `react-diff-viewer-continued` renders text safely |
| T-05-04 | Spoofing / Elevation | Unauthenticated access to the review/approve/reject endpoints and the pull API | mitigate | `require_api_key` dependency on both routers rejects a missing/wrong `X-API-Key` with 401 (review finding #12); satisfies ASVS L1 V4 access control for an internal Phase-1 tool; full per-tenant auth is Phase 8 (AUDIT-02) |
| T-05-05 | Tampering | CSRF on approve/edit/reject mutations | accept | The review API uses a header API key, not a cookie session — there is no ambient credential a CSRF attack could ride; documented as accepted for the internal-only Phase 1 surface |
| T-05-06 | Spoofing / Repudiation | Reviewer identity is a free-text field, not authenticated | accept | Per CONTEXT.md Claude's-discretion and Deferred Ideas — full reviewer attribution / audit trail is explicitly Phase 8 (AUDIT-02). Phase 1 captures `reviewer_name` as a free-text field only; the API key authenticates the tool, not the individual reviewer; accepted for an internal tool |
| T-05-07 | Tampering | Illegal lifecycle transition / stale concurrent action via a crafted or racing API call | mitigate | approve/edit/reject call `assert_transition`; mutating endpoints load the Change `FOR UPDATE`; `IllegalTransitionError` maps to HTTP 409 — illegal and stale-concurrent transitions are rejected, not silently applied (review finding #11) |
| T-05-08 | Information Disclosure | Overly-broad CORS exposing the API to arbitrary origins | mitigate | CORS allows only the `http://localhost:5173` Vite dev origin — not `*` |
</threat_model>
